This guide focuses on the mandatory implementation details reviewers need to understand: CLI generation, metadata scanning, frontend-driven filter definitions, URL and tree filters, dependency injection, query construction, joins, grouping, and the request/response flow.

1. Architecture at review level

Configuration

Defines import paths, generated locations, anchors, enabled layers, scan depth, and safety behavior.

Scanner

Model graph discovery

Generated backend

Endpoint code path 
@router.get("/{entity}/query")
def query_entity(
    entity: str,
    request: Request,
    page: int = Query(1, ge=1),
    size: int = Query(20, ge=1, le=100),
    sort_by: Optional[str] = Query(None),
    order: str = Query("asc", pattern="^(asc|desc)$"),
    search: Optional[str] = Query(None),
    db: Any = Depends(get_db),
) -> dict[str, Any]:
    entity_meta = _get_entity(registry, entity)
    model = _model_for_entity(entity_meta)
    filters = _parse_url_filters(request)
    return _run_query(
        db, entity_meta, model,
        page=page, size=size,
        sort_by=sort_by, order=order,
        search=search, filters=filters,
        filter_tree=None,
    )

What this means in plain language

  1. The URL says which entity to query: companies, departments, tasks, and so on.
  2. The router looks up that entity in the generated registry instead of using hardcoded routes.
  3. The registry tells it the Python module for the real SQLAlchemy model.
  4. FastAPI injects the database session through Depends(get_db).
  5. The request query string is converted into filter rules.
  6. _run_query() performs the actual SQLAlchemy query pipeline.
Query pipeline 
def _run_query(db, entity, model, *, page, size,
               sort_by, order, search, filters, filter_tree):
    query = db.query(model)

    for filter_rule in filters:
        query, _ = _apply_filter(query, model, filter_rule)

    query, _ = _apply_tree(query, model, filter_tree)
    query = _apply_search(query, model, entity, search)

    total = _count_query(query, model)
    query = _apply_sort(query, model, entity, sort_by, order)

    rows = query.offset((page - 1) * size).limit(size).all()
    return { "data": serialize(rows), "meta": page_meta }

Why the order matters

  • Filters run before count so pagination metadata reflects the filtered dataset.
  • Search runs before count for the same reason.
  • Sorting happens before offset/limit so each page is deterministic.
  • Serialization happens last so SQLAlchemy does the database work first.

3. Filter tree system demystified

The tree system exists because real users need filters more complex than a flat list. A tree lets the UI express nested logic: (A AND B) OR (C AND D). The backend recursively evaluates the tree into SQLAlchemy boolean expressions.

Condition node

A leaf. It maps one field path, one operation, and one value into a SQLAlchemy predicate. 

{
  "node_type": "condition",
  "field": "company.name",
  "operation": "ilike",
  "value": "acme"
}

Operator node

A group. It combines child conditions or child groups with AND or OR. 

{
  "node_type": "operator",
  "operator": "AND",
  "children": [
    { "node_type": "condition", "field": "budget", "operation": "gte", "value": 1000 },
    { "node_type": "condition", "field": "company.name", "operation": "ilike", "value": "acme" }
  ]
}

Backend evaluation rules

Step What happens Why it matters
Validate node shape Each node must be a condition or an operator. Prevents ambiguous filter input from reaching SQL construction.
Resolve field path company.name becomes a relationship join to company and a column access to name. The UI can filter relationships without custom endpoints.
Build predicate ilike becomes column.ilike("%value%"). Operations stay consistent across all entities.
Combine clauses Child clauses are combined with SQLAlchemy and_ or or_. Nested business filters are represented exactly.
Depth guard The generated runtime limits tree depth. Protects the API from pathological or accidental recursive filters.
Tree evaluator code 
def _evaluate_tree(query, model, node, depth=0):
    if depth > MAX_FILTER_TREE_DEPTH:
        raise HTTPException(400, "Filter tree too deep")

    if node["node_type"] == "condition":
        column, query, joined = _resolve_field(
            query, model, node["field"], node["operation"]
        )
        clause = _filter_expression(
            column, node["operation"], node.get("value")
        )
        return query, clause, joined

    clauses = []
    for child in node["children"]:
        query, child_clause, _ = _evaluate_tree(query, model, child, depth + 1)
        clauses.append(child_clause)

    if node["operator"] == "AND":
        return query, and_(*clauses), True
    return query, or_(*clauses), True

Mental model for reviewers

Think of the JSON tree as a safe, structured way to build a SQL WHERE clause.

  • A condition node becomes one SQL predicate.
  • An operator node becomes a parenthesized group of predicates.
  • Nested operators become nested parentheses.
  • The backend never executes raw user SQL; it maps known operations to SQLAlchemy expressions.
Operation mapping 
def _filter_expression(column, operation, value):
    if operation == "eq":    return column == value
    if operation == "ne":    return column != value
    if operation == "gte":   return column >= value
    if operation == "ilike": return column.ilike(f"%{value}%")
    if operation == "in":    return column.in_(value)
    if operation == "between":
        return column.between(value[0], value[1])

Why this is review-critical

This is where user intent becomes a database predicate. Reviewers should verify three things:

  1. Only supported operation names are accepted.
  2. Operations validate value shape, especially in and between.
  3. SQLAlchemy expressions are used instead of string-built SQL.

4. Odoo-style searching and field-type input

FilterX supports quick filters through URL parameters. The pattern is simple: field_operation=value. If no operation suffix is provided, the default operation is eq. 

# String contains search
GET /api/filterx/companies/query?name_ilike=acme

# Numeric range-style comparison
GET /api/filterx/projects/query?budget_gte=10000&budget_lte=50000

# Relationship path filter
GET /api/filterx/employees/query?department.company.name_ilike=acme

Field type controls frontend input

Field type Typical UI input Allowed operations
string, text, enum Text input or select eq, ne, like, ilike, starts_with, ends_with, in, not_in, null checks
integer, float Number input, range input eq, ne, comparisons, between, in, not_in, null checks
boolean Toggle or tri-state select eq, ne, null checks
date, datetime Date picker or range picker eq, ne, comparisons, between, null checks
json Advanced / restricted input eq, ne, null checks
Why this matters: The frontend should not guess which operations are valid. It reads field metadata from /api/filterx/metadata and renders only the inputs and operators that make sense for that field type.
URL grammar parser 
VALID_OPS = { "eq", "ne", "gte", "ilike", "in", "between", ... }

def _parse_filter_param(key: str, value: str):
    for op in sorted(VALID_OPS, key=len, reverse=True):
        suffix = f"_{op}"
        if key.endswith(suffix):
            field = key[:-len(suffix)]
            return {
                "field": field,
                "operation": op,
                "value": _parse_value(value, op),
            }

    # No suffix means equality
    return { "field": key, "operation": "eq", "value": _parse_value(value, "eq") }

Human explanation

The parser turns a query string into the same internal filter format used by the tree system.

  • name_ilike=acme becomes field name, operation ilike, value acme.
  • budget_gte=1000 becomes field budget, operation gte, numeric value 1000.
  • is_active=true becomes equality with boolean true.
  • department.company.name_ilike=acme remains a dot-path field and is resolved later by join logic.
Metadata-driven frontend input 
// Example metadata field returned to the UI
{
  "name": "budget",
  "type": "float",
  "ops": ["eq", "ne", "gt", "gte", "lt", "lte", "between"]
}

// UI behavior
field.type === "float"    → show number input
field.ops includes "between" → allow min/max range
field.ops excludes "ilike"   → do not show text contains operator

Why this matters to reviewers

The frontend is not an independent source of truth. It is a renderer of the backend metadata contract.

That is what makes the UI generic: when a new model is scanned, the UI can build the correct filter controls without a hand-written config file for that model.

5. Dependency injection and query execution

The generated router does not create database sessions manually. It imports the configured session dependency from python.session_dependency_import, then uses FastAPI Depends. This keeps FilterX aligned with the host project's transaction/session lifecycle. 

def create_router(api_prefix="/api", entities=None, session_dependency_import="app.database:get_db"):
    get_db = _import_object(session_dependency_import)

    @router.get("/{entity}/query")
    def query_entity(..., db = Depends(get_db)):
        entity_meta = _get_entity(registry, entity)
        model = _model_for_entity(entity_meta)
        return _run_query(db, entity_meta, model, ...)
Host project contract 
# app/database.py in the host project
engine = create_engine(DATABASE_URL)
SessionLocal = sessionmaker(bind=engine)
Base = declarative_base()

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

# filterx.yaml points to it
python:
  session_dependency_import: app.database:get_db

What FilterX assumes

FilterX only assumes that the host exposes a function compatible with FastAPI dependencies.

  • The host decides how sessions are created.
  • The host decides when sessions are closed.
  • FilterX simply asks FastAPI to inject db for each request.

Why dependency injection is mandatory

Project compatibility

Each host project may use different session setup, middleware, or transaction rules.

Testability

FastAPI dependency overrides can replace the database during tests.

No global coupling

Generated code uses the configured import path rather than hardcoding a session factory.

6. Joins and relationship paths

Joins are driven by dot notation. A path like department.company.name is resolved one segment at a time: each middle segment must be a SQLAlchemy relationship, and the final segment must be a column.

Start from the root model

For /api/filterx/employees/query, the root model is Employee.

Walk relationship segments

department joins Employee.department. company joins Department.company.

Resolve the final field

name becomes the column used in the predicate, sort, or group-by expression.

Apply SQL expression

The generated runtime applies the operation, such as company.name ILIKE '%acme%'. 

# Multi-hop relationship query
GET /api/filterx/employees/query?department.company.name_ilike=acme

# Relationship group-by
GET /api/filterx/employees/group-by/department.company.name
Relationship resolver code 
def _resolve_field(query, root_model, field_path, operation=None):
    parts = field_path.split(".")
    current = root_model

    for rel_name in parts[:-1]:
        rel_attr = getattr(current, rel_name)
        related_alias = aliased(rel_attr.property.mapper.class_)
        query = query.join(related_alias, rel_attr)
        current = related_alias

    column_name = parts[-1]
    return getattr(current, column_name), query, True

Human explanation

Each dot segment before the last one is treated as a relationship to join through.

  1. department.company.name is split into department, company, and name.
  2. The backend starts at the requested root model, for example Employee.
  3. It joins from Employee to Department.
  4. Then it joins from Department to Company.
  5. The final part, name, becomes the column used in the filter, sort, or group-by.
Why aliases are used 
related_alias = aliased(rel_attr.property.mapper.class_)
query = query.join(related_alias, rel_attr)
current = related_alias

Reviewer note

Aliases make joins safer when the same table can appear more than once, especially with self-relations like Employee.manager.

Without aliases, SQLAlchemy can produce ambiguous SQL when a model joins to itself or when multiple paths reach the same target table.

Important review note: The generated code supports arbitrary dot paths that SQLAlchemy can resolve. Practical limits should still be controlled by scan depth, UI choices, indexing, and project-specific performance policies.

7. Grouping flow

Grouping is server-side. The backend applies any filters first, resolves the group field, then emits GROUP BY with COUNT. The frontend receives a simple analytics-ready array. 

# Request
GET /api/filterx/tasks/group-by/status

# Response
[
  { "key": "open", "count": 12 },
  { "key": "done", "count": 7 }
]

Filtered grouping

POST /api/filterx/tasks/group-by/status/filter

{
  "node_type": "condition",
  "field": "project.department.company.name",
  "operation": "ilike",
  "value": "acme"
}
Grouping implementation 
def _run_group_by(db, entity, model, *, field, search, filters, filter_tree):
    query = db.query(model)

    for filter_rule in filters:
        query, _ = _apply_filter(query, model, filter_rule)

    query, _ = _apply_tree(query, model, filter_tree)
    query = _apply_search(query, model, entity, search)

    column, query, _ = _resolve_field(query, model, field)
    rows = (
        query.with_entities(column, func.count().label("count"))
             .group_by(column)
             .order_by(func.count().desc())
             .all()
    )
    return [{ "key": row[0], "count": row[1] } for row in rows]

Human explanation

Grouping is not a separate query language. It reuses the same filtering machinery first, then groups what remains.

  1. Start with the entity table.
  2. Apply URL filters, tree filters, and search.
  3. Resolve the group-by field, including relationship paths.
  4. Replace normal row selection with field, count(*).
  5. Return a small array the frontend can use for chips, charts, facets, or summaries.

8. What makes it generic

No hardcoded entity routers

The same generated endpoint receives an {entity} path parameter and resolves it against the scanned registry.

No hand-written response schemas required

The runtime serializes ORM columns generically and includes shallow many-to-one relationships for common table rendering.

Runtime model import

Each entity stores its original module path, so the router can import the correct model class dynamically.

Field-type driven behavior

The scanner maps SQLAlchemy types to filter types and valid operations, allowing the UI and backend to share the same contract.

9. Critical CLI parts and why they exist

CLI part Mandatory role Why it exists
filterx.yaml Contract between FilterX and the host project. Prevents assumptions about project structure. The project tells FilterX where the app, Base, models, session dependency, and anchors live.
scan Builds .filterx/scan.json, diagnostics, and plan. Generation must be based on real model metadata, not static templates.
backend install Generates runtime query router and mounts it. This is the core integration: it turns scanned models into a usable API.
frontend install Generates frontend entity config and optional routes. Lets UI screens consume the same metadata contract instead of redefining fields manually.

rollback Restores patch bundle backups. Generated code must be reversible for safe adoption in existing projects.

Manifest and patches Tracks written files and backups. Supports idempotency, conflict handling, and rollback.
Scanner core 
def run_scan(config, project_root):
    sys.path.insert(0, str(project_root))

    _walk_and_import_package(config["python"]["models_package"])
    base_cls = _import_object(config["python"]["base_class_import"])

    mapper_classes = [m.class_ for m in base_cls.registry.mappers]
    entities = [_build_entity(cls) for cls in mapper_classes]

    return {
      "entities": entities,
      "relationship_graph": graph,
      "routes": route_scan,
      "diagnostics": diagnostics,
    }

Why scan must exist before generation

The generator should never invent model knowledge. It must read the host project first.

  • models_package decides which models are discoverable.
  • Base.registry.mappers is the source of ORM truth.
  • Diagnostics expose risky model graph facts like cycles, deep joins, and composite keys.
  • The generated API is only as accurate as the scan artifact.
Backend generation plan 
def _build_patch_ops(generated_root, entities, api_prefix, session_dependency_import, ...):
    return [
        PatchOp("generated_file", "app/filterx_generated/entities.py", _render_entities_py(entities)),
        PatchOp("generated_file", "app/filterx_generated/router_factory.py", _render_router_factory_py()),
        PatchOp("generated_file", "app/filterx_generated/router.py", _render_router_py(api_prefix, session_dependency_import)),
        PatchOp("anchor_insert", "app/main.py", snippet="app.include_router(...)"),
    ]

Why generated files are patch operations

FilterX treats every write as an operation. That allows dry-run, validation, idempotency, and rollback to all work from the same plan.

  1. Generated files are owned by FilterX.
  2. Host files are only touched through anchor inserts.
  3. The patcher can preview, apply, skip unchanged files, or roll back.
Safety preflight

Why reviewers should care

Generated code is powerful, so the CLI blocks unsafe installs before writing files.

  • If the route already exists, generation is stopped.
  • If the anchor is missing, generation is stopped.
  • If generated files changed and overwrites are disabled, generation is stopped.

10. Code review checklist

Backend runtime

  • Unknown entity returns 404.
  • Invalid field path returns 400.
  • Invalid operation returns 400.
  • Pagination uses offset and limit.
  • Joined counts use distinct root primary key.
  • Tree depth is guarded.
  • Session is injected through the configured dependency.

CLI safety

  • Dry-run produces no writes.
  • Missing anchors block installation in strict mode.
  • Generated files are idempotent.
  • Route conflicts are detected before writes.
  • Rollback restores or removes touched files.
  • Scan diagnostics reveal cycles, composite PKs, and depth issues.
