FilterX Functional Guide

Plug a complete filtering API into any FastAPI + SQLAlchemy project.

FilterX is a CLI-driven integration system that scans existing SQLAlchemy models, generates a reusable query API, mounts it safely into FastAPI, and gives frontend teams metadata to build advanced filtering screens without rewriting model-specific filtering code for every entity.

1 CLIScan, generate, validate, rollback.
Any model graphFields, relationships, joins, and types are discovered.
Real APIQuery, filter trees, sorting, pagination, search, grouping.

The problem

Repeated work

Every list page repeats the same backend logic

Teams write one-off endpoints for pagination, search, sorting, filters, joins, and grouping for every model.

Drift

Frontend and backend definitions go out of sync

Fields, filter operators, types, and relationships are duplicated in multiple layers and eventually diverge.

Risk

Manual integration is fragile

Blind code edits and copy-pasted routers make adoption risky in existing projects.

The FilterX answer

Generate from your real models

FilterX imports the configured SQLAlchemy model package, reads columns and relationships, and builds an entity registry.

  • Field names and SQL types
  • Permitted operations by type
  • Primary keys and foreign keys
  • Relationship paths for joins

Patch only where explicitly allowed

Host files are modified only at anchor comments. No blind rewrite, no guessing, and every write has rollback metadata. 

# In app/main.py
# FILTERX:ROUTER_MOUNT

What you get after integration

Endpoint Purpose
GET /api/filterx/metadata Frontend discovers entities, fields, relationship metadata, and valid operators.
GET /api/filterx/{entity}/query Odoo-style URL search with pagination, sorting, text search, and field filters.
POST /api/filterx/{entity}/filter Advanced custom filter trees with nested AND / OR conditions.
GET /api/filterx/{entity}/group-by/{field} Server-side counts by field or relationship field.
POST /api/filterx/{entity}/group-by/{field}/filter Grouped analytics after applying an advanced filter tree.

Why the CLI exists

Repeatable adoption

The same workflow works for a demo app, an ERP, or a new customer project because the integration contract is declarative.

Safe operations

Dry-run, diagnostics, manifest tracking, conflict detection, and rollback reduce the risk of introducing generated code into an existing codebase.

Integration guide for the dummy project

Target directory: Run the following commands from filterx_robustness_lab.

Move into the project

This makes every relative path in filterx.yaml resolve correctly. 

Set-Location "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\filterx_robustness_lab"

Install the target app and local CLI

This installs the dummy app dependencies and the editable FilterX CLI from the local source project. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\python.exe" -m pip install -e .
& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\python.exe" -m pip install -e "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\filter_test_project\tools\filterx"

Prepare the database

FilterX itself does not require a DB during scanning, but runtime testing needs actual tables and rows. 

Copy-Item .env.example .env -Force
# Confirm DATABASE_URL in .env points to PostgreSQL:
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/filterx_robustness_lab

Create the database once in PostgreSQL:

CREATE DATABASE filterx_robustness_lab;

Create domain tables

This creates the normal business tables for companies, departments, employees, projects, assignments, and tasks. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\alembic.exe" revision --autogenerate -m "init_domain_models"
& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\alembic.exe" upgrade head

Seed test data

This gives the generated filter endpoints real relational data to query. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\python.exe" seed_data.py

Scan the project

FilterX imports the configured models package and writes the model graph into .filterx. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\filterx.exe" scan --project-root . --config filterx.yaml --no-dry-run --json

Expected: 6 entities. Warnings about relationship cycles and composite keys are acceptable in this robustness lab.

Generate the backend API

This creates app/filterx_generated and mounts the router at the configured anchor. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\filterx.exe" backend install --project-root . --config filterx.yaml --no-dry-run --yes --json

Validate the generated backend

This confirms generated files exist and the mount snippet is present. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\filterx.exe" backend validate --project-root . --config filterx.yaml --json

Expected: error_count: 0 and warning_count: 0.

Run the backend

The generated API is now available under /api/filterx. 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\uvicorn.exe" app.main:app --reload

Why a project should adopt it

Faster delivery

New list screens get filtering, sorting, pagination, grouping, and search without writing custom query endpoints per model.

Consistent behavior

Every entity uses the same API contract, response shape, operator names, and filter tree language.

Safer integration

The CLI makes changes visible, reversible, and anchored to known host files.

Rollback if needed

Every write is backed by patch metadata. If a generated integration needs to be reversed: 

& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\filterx.exe" rollback --project-root . --config filterx.yaml --list
& "c:\Users\raedn\OneDrive\Bureau\FILES\DEV FILES\MAIN_PROJECTS\.venv\Scripts\filterx.exe" rollback --project-root . --config filterx.yaml
